L — Local Primitives  ·  in-process concurrency
🔒
Mutex (Mutual Exclusion Lock)
One owner at a time — the fundamental building block
single-process blocking
How it works
  • Binary state: locked or unlocked
  • Only the thread that acquired it can release it (ownership)
  • All other threads block (sleep) in a wait queue until freed
  • OS scheduler wakes a waiting thread on release
  • Recursive variant allows same thread to re-acquire without deadlock
When to use
  • Protecting shared mutable state (counters, maps, caches)
  • Short critical sections with low contention
  • Any time you need exclusive write access
  • Connection pool slot management
  • Avoid: long I/O or DB calls inside the critical section
GranularityExclusive
BlockingYes (sleep)
FairnessOS-dependent
OverheadLow–Medium
Go · sync.Mutex
var mu      sync.Mutex
var counter int

func increment() {
    mu.Lock()
    defer mu.Unlock()  // always defer — never forget to unlock
    counter++
}
⚡
Classic deadlock: lock inversion. Thread A holds Lock 1, waits for Lock 2. Thread B holds Lock 2, waits for Lock 1. Fix: always acquire multiple locks in a consistent global order across your entire codebase.
📖
Read-Write Lock (RWMutex)
Many concurrent readers OR one exclusive writer — never both
single-process read-optimized
How it works
  • Multiple goroutines/threads can hold a read lock simultaneously
  • A write lock is exclusive — blocks all readers and writers
  • New readers are blocked once a writer is waiting (writer preference)
  • Writers wait until all active readers release
  • Internally tracks a reader count + a writer-pending flag
When to use
  • Read-heavy workloads: config maps, in-memory caches, routing tables
  • Writes are rare — e.g. config reload once per minute
  • Rule of thumb: reads > 80% of all accesses
  • Not worth it when reads and writes are balanced — plain Mutex is simpler
Read concurrencyUnlimited
Write concurrencyExclusive
Write starvationPossible
Overhead vs MutexHigher
Go · sync.RWMutex
var rw    sync.RWMutex
var cache map[string]string

func read(key string) string {
    rw.RLock()
    defer rw.RUnlock()
    return cache[key]
}
func write(key, val string) {
    rw.Lock()
    defer rw.Unlock()
    cache[key] = val
}
⚠️
Never upgrade an RLock to a full Lock in the same goroutine — that's an instant deadlock. Also: continuous reads can starve writers indefinitely. Most runtimes (Go, Java) implement writer-preference to prevent this, but verify your platform's behavior.
🌀
Spinlock
Busy-wait in a tight loop — no sleep, no context switch
single-process CPU-burning
How it works
  • Uses an atomic CAS (compare-and-swap) instruction in a loop
  • Thread repeatedly checks the flag — never yields the CPU
  • No OS involvement; no thread context switch overhead (~1–10µs saved)
  • If lock is free → acquired in nanoseconds
  • If lock is held → burns CPU until it's released
When to use
  • Critical section is extremely short (< ~100ns)
  • OS context switch cost (>1µs) would dominate the actual work
  • Kernel code, interrupt handlers, lock-free data structures internals
  • Never in user-space with long or I/O critical sections
  • Never on a single-CPU system — will livelock forever
Latency~nanoseconds
CPU while waiting100%
Context switchNone
Best forKernel / RT
Go · atomic spinlock pattern
import "sync/atomic"

type SpinLock struct{ state int32 }

func (s *SpinLock) Lock() {
    for !atomic.CompareAndSwapInt32(&s.state, 0, 1) {
        runtime.Gosched()  // yield to avoid starvation on cooperative schedulers
    }
}
func (s *SpinLock) Unlock() { atomic.StoreInt32(&s.state, 0) }
🔥
Adaptive spinlocks (Linux kernel, JVM HotSpot) spin briefly then fall back to sleeping. This gets the best of both worlds: nanosecond acquisition when uncontended, no CPU waste under load. Java's synchronized does this automatically via biased locking → thin lock → inflated lock progression.
O — Ownership Model  ·  counting & reentrance
🎛️
Semaphore
Counting lock — controls access to a pool of N resources
single-process counting
How it works
  • Maintains an integer counter N (initial capacity)
  • Acquire / Wait (P): decrements counter; blocks if counter = 0
  • Release / Signal (V): increments counter; wakes a blocked goroutine
  • Binary semaphore (N=1) ≈ Mutex but without ownership — any thread can release
  • Counting semaphore (N>1) allows exactly N concurrent accesses
When to use
  • Limiting concurrent DB connections or HTTP workers to exactly N
  • Rate-limiting goroutines in a bounded worker pool
  • Producer-consumer synchronization (buffer slot counting)
  • Throttling outbound calls to a third-party rate-limited API
  • Key insight: no ownership makes cross-goroutine signaling natural
Max concurrentN (configurable)
OwnershipNone
Use caseResource pool
BlockingYes (sleep)
Go · semaphore via buffered channel
// Limit to 10 concurrent DB connections
sem := make(chan struct{}, 10)

func queryDB() {
    sem <- struct{}{}         // acquire a slot
    defer func() { <-sem }()  // release on return
    // ... do the query
}
💡
Semaphore ≠ Mutex. The key interview distinction: "Can the releaser be a different goroutine than the acquirer?" If yes → semaphore. If no → mutex. Semaphores model resource availability; mutexes model data ownership. Confusing them is a classic concurrency bug.
🔁
Reentrant Lock (Recursive Mutex)
Same thread can lock again without deadlocking itself
single-process recursive
How it works
  • Tracks the owning thread ID + a recursion depth counter
  • If the same thread calls Lock() again → depth increments, no block
  • Unlock() decrements depth; lock fully releases only when depth hits 0
  • A different thread still blocks normally
  • Java's synchronized keyword and ReentrantLock work this way
When to use
  • Recursive algorithms that internally call other locked methods
  • OO code where a public method calls another locked method on the same object
  • Legacy codebases where lock boundaries are tangled
  • Caution: often masks design issues — prefer refactoring
  • Go deliberately omits this — treats it as an anti-pattern signal
Java · ReentrantLock — recursive acquisition
ReentrantLock lock = new ReentrantLock();

void outer() {
    lock.lock();              // depth: 1
    try { inner(); }          // safe — same thread re-acquires
    finally { lock.unlock(); } // depth: 1 → 0 → released
}
void inner() {
    lock.lock();              // depth: 2
    try { /* work */ }
    finally { lock.unlock(); } // depth: 2 → 1
}
⚠️
Go intentionally has no reentrant mutex. If you find yourself needing one, it's a strong signal that your locking granularity is too coarse or your call graph is tangled. Prefer passing unlocked data down the call stack, or splitting critical sections into smaller pieces.
C — Contention Strategy  ·  optimistic vs pessimistic
🎯
Optimistic Locking (CAS / MVCC)
Assume no conflict — validate at commit, retry if wrong
in-process & DB non-blocking
How it works
  • Read: fetch the data + current version/timestamp
  • Modify: compute new value locally — no lock held during this
  • Validate + Write: atomically check version hasn't changed, then commit
  • If version changed → conflict → abort and retry from the read step
  • In-process: sync/atomic CAS. In DB: version column or MVCC snapshots
When to use
  • Low-contention workloads — conflicts are the exception
  • Read-heavy; writes are occasional and quick
  • Lock-free counters, accumulators, atomic flags in hot paths
  • DB row updates where concurrent edits are uncommon
  • Avoid: high-contention write storms — retry loops thrash CPU and DB
BlockingNone
On conflictRetry
Low contentionHigh throughput
High contentionDegrades badly
Go · CAS loop  |  SQL · version column
// In-process: atomic CAS loop
for {
    old := atomic.LoadInt64(&val)
    if atomic.CompareAndSwapInt64(&val, old, old+1) { break }
    // conflict — loop and retry
}

-- DB: version column optimistic lock
UPDATE orders SET status='shipped', version=version+1
WHERE id=42 AND version=7;
-- 0 rows affected = conflict → re-read and retry
💡
MVCC in Postgres is optimistic locking at the engine level. Readers never block writers; each transaction sees a consistent snapshot. This is why Postgres reads scale so well — no read lock to acquire. The tradeoff: vacuum overhead from accumulating old row versions (dead tuples).
🛡️
Pessimistic Locking
Assume conflict — lock before reading, hold until done
DB / in-process blocking
How it works
  • Acquire an exclusive lock before reading the data
  • Hold the lock through the entire read-modify-write cycle
  • Other transactions block until the lock is released on COMMIT/ROLLBACK
  • In SQL: SELECT ... FOR UPDATE acquires a row-level exclusive lock
  • Conflict is impossible — serialization is guaranteed
When to use
  • High-contention rows — conflicts are frequent
  • Inventory / seat booking — must prevent oversell
  • Financial ledger updates — correctness over throughput
  • Long read-modify-write cycles where retries are expensive
  • Avoid: high-read rows — blocks every concurrent reader
Conflict riskEliminated
ConcurrencyReduced
Deadlock riskYes
Retry neededNo
SQL · SELECT FOR UPDATE — row-level exclusive lock
BEGIN;

-- Acquire row-level exclusive lock immediately
SELECT quantity FROM inventory
WHERE sku = 'ABC' FOR UPDATE;   -- blocks concurrent writers

-- Safe to modify — no race condition possible
UPDATE inventory
   SET quantity = quantity - 1
 WHERE sku = 'ABC';

COMMIT;  -- lock released here
⚡
DB deadlock detection is automatic but expensive. Postgres detects deadlocks via wait-for graph cycle detection and kills one transaction. Always catch deadlock errors (error code 40P01) and retry the entire transaction. Keep transactions short and lock rows in a consistent order to minimize deadlock frequency.
K — Koordinated  ·  distributed locking across nodes
🗝️
Distributed Lock — Redis SET NX
Single-node distributed mutex via atomic Redis commands
distributed single-node
How it works
  • Atomic SET key value NX PX ttl — sets only if Not eXists
  • Value is a unique token (UUID) — proves ownership on release
  • TTL is mandatory — auto-expires stale locks if holder crashes
  • Release: Lua script checks token matches, then DELetes atomically
  • Only safe with a single Redis primary — no failover guarantees
When to use
  • Preventing duplicate job execution across worker pods
  • Rate limiting per user across multiple API servers
  • Leader election where Redis downtime is acceptable
  • Short-duration locks on non-critical resources
  • Not for: anything requiring safety guarantees on Redis failover
Fault toleranceLow (1 node)
Latency~1ms
ComplexityLow
CorrectnessBest-effort
Redis · acquire + release with Lua
# Acquire (NX = only if not exists, PX = TTL in ms)
SET lock:job:42 "uuid-abc-123" NX PX 30000
# Returns OK on success, nil if already locked

# Release: Lua script — check + delete atomically
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0  -- not the owner, refuse to delete
end
🚨
Never DEL without verifying ownership. If you skip the Lua check, you might delete a lock acquired by another process after yours expired. The UUID token + Lua atomicity pattern is mandatory, not optional. Without it you have a race condition on the release path.
🔴
Redlock Algorithm
Fault-tolerant distributed lock across N independent Redis nodes
distributed multi-node
How it works
  • Requires N independent Redis nodes (typically 5, no replication between them)
  • Client tries SET NX on all N nodes sequentially, recording elapsed time
  • Lock succeeds only if acquired on majority (⌊N/2⌋ + 1) nodes
  • Valid TTL = original TTL − time spent acquiring
  • On any failure: release all acquired nodes and abort
When to use / avoid
  • Distributed mutex resilient to single Redis node failure
  • Acceptable: slightly higher latency for stronger availability
  • Controversy: Martin Kleppmann showed GC pauses + clock skew can still violate safety — fencing tokens are needed for true correctness
  • For strong guarantees: use etcd or ZooKeeper instead
Nodes needed5 (typical)
Fault tolerance⌊N/2⌋ failures
Latency~5ms
CorrectnessProbabilistic
🎓
The fencing token pattern solves the GC pause problem. Even with Redlock, a GC pause can make a process hold a lock past its TTL while believing it's still valid. Fix: lock service returns a monotonically increasing fencing token with each grant. The storage backend rejects any write carrying an older token. Safe even under clock skew.
🌳
ZooKeeper / etcd Consensus Locks
Strongly consistent distributed locks via Raft or ZAB consensus
distributed CP / strong
How it works (etcd)
  • Creates an ephemeral key with a lease TTL tied to a client session
  • Uses Raft consensus — majority quorum required before any write commits
  • Key auto-deleted if client crashes and fails to renew the lease
  • Monotonic revision numbers serve as built-in fencing tokens
  • ZooKeeper: ephemeral sequential znodes + Watcher mechanism for fair queuing
When to use
  • Leader election in distributed services (used by Kubernetes itself)
  • Cluster membership and service discovery
  • Config distribution requiring strong consistency
  • When correctness is non-negotiable (financial systems, infrastructure)
  • Accept: higher latency (10–50ms) and operational complexity vs Redis
ConsistencyStrong (CP)
Latency10–50ms
Fault tolerance⌊N/2⌋ failures
FencingBuilt-in
etcd · lease-based distributed lock (Go)
// Grant a 30s lease tied to this client session
lease, _ := client.Grant(ctx, 30)

// Atomic create-if-not-exists via transaction
txn := client.Txn(ctx).If(
    clientv3.Compare(clientv3.CreateRevision("lock/job"), "=", 0),
).Then(
    clientv3.OpPut("lock/job", nodeID, clientv3.WithLease(lease.ID)),
)
resp, _ := txn.Commit()
// resp.Revision is your fencing token — use it on writes
🏗️
etcd powers Kubernetes control plane. Every etcd write goes through Raft — the cluster won't acknowledge until a quorum agrees. This is why etcd revision numbers are safe fencing tokens. For most application-level locking, Redis SET NX is sufficient. Reach for etcd when you're building infrastructure or coordination services.
S — Senior / Staff Insights  ·  tradeoffs & real-world pitfalls
01
Lock Granularity Is Everything
A coarse lock (one mutex for the whole cache) is simple but serializes all access. Fine-grained locks (per-key shards) scale but multiply deadlock surface area. Stripe your locks — shard a data structure into N buckets each with its own mutex. Java's ConcurrentHashMap uses 16 stripes by default. The ideal granularity minimizes contention without multiplying lock-ordering bugs. When in doubt, profile first.
02
Lock-Free > Locked When Possible
Atomic operations (sync/atomic, java.util.concurrent.atomic) eliminate lock overhead entirely for counters, flags, and pointers. Rule: if you only need to atomically update a single value, reach for atomics first. Locks are for multi-step invariants where you need several fields to change together consistently. Lock-free ring buffers and queues can push throughput 5–10× vs mutex-based equivalents.
03
Distributed Locks Are a Last Resort
Distributed locks add latency, introduce a new failure mode (lock service outage), and don't compose with DB transactions. Before reaching for Redis SET NX, ask: can I use DB row-level locking? Can I partition work so each node owns its slice of data exclusively? Idempotency + at-least-once delivery often eliminates the need for distributed locks entirely. Design for idempotency first — it's more resilient and simpler to reason about.
04
The Thundering Herd Problem
When a popular cache key expires, thousands of requests simultaneously find a cache miss and stampede the DB. Solutions: probabilistic early expiration (randomly refresh before TTL), mutex-protected cache fill (only one goroutine rebuilds; others wait on the same future), or stale-while-revalidate (serve stale data while one worker refreshes async). Go's singleflight package is the canonical fix — it collapses concurrent identical requests into one.
05
Clock Skew Breaks Naive Distributed Locks
TTL-based distributed locks assume synchronized clocks. In practice, NTP drift, GC pauses (stop-the-world can be hundreds of ms), or VM live-migration can cause a process to hold a lock past its TTL while thinking it's still valid. Always pair distributed locks with fencing tokens — monotonically increasing values the storage backend uses to reject out-of-order writes. Never use wall-clock timestamps as validity checks.
06
Profile Contention Before Optimizing
Before switching Mutex → RWMutex or sharding, measure first. Go's pprof mutex profile shows exact contention points. Java has -XX:+PrintConcurrentLocks. Most contention is in 1–2 hot paths. Switching from pessimistic to optimistic locking on a high-contention row can actually make things worse — more retries, more wasted work, higher CPU. Profile, don't guess.
07
Interview Signal: Frame Every Lock Decision as a Tradeoff
Strong candidates don't just name the right lock — they articulate the tradeoff clearly. "I'd use optimistic locking here because reads dominate and retries are cheap. If contention were high or the write cycle long, I'd switch to pessimistic to avoid wasted retry work." For distributed: "Redis SET NX gives us sub-millisecond latency and simplicity, but only best-effort safety. If this is a financial ledger or infrastructure coordination, I'd pay the 10–50ms for etcd's strong consistency and built-in fencing tokens." The ability to move fluidly between lock types based on workload characteristics — and explain why — is the mark of a senior engineer.
