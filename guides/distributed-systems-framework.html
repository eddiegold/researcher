System Design Interview Framework
RADIO
A 5-step mental model for designing distributed systems in 60 minutes
0
60 min
R
Requirements & Constraints
Nail the problem before touching the solution
~8 min
Functional
  • What are the core user-facing features? (list 3–5 max)
  • What does each feature need to do, end-to-end?
  • Who are the actors and what actions do they take?
  • Any out-of-scope features to explicitly set aside?
Non-Functional
  • Scale: DAU, requests/sec, data volume
  • Latency: p99 read/write targets (ms)
  • Availability: 99.9% vs 99.99%? Tolerate stale data?
  • Consistency: Strong, eventual, or read-your-writes?
  • Durability: Can we lose writes?
⚡
Drive the conversation. Don't wait for the interviewer to hand you requirements — ask targeted questions and write the constraints on the board. This signals engineering maturity. The most important trade-off to surface early: read-heavy vs write-heavy and consistency vs availability.
A
API Design
Define contracts before internals
~8 min
Sketch the surface area
  • Enumerate one endpoint per core feature (method, path, key params)
  • Identify request/response payloads — just the critical fields
  • Note pagination strategy (cursor vs offset)
  • Flag any streaming or async patterns needed
Key decisions
  • REST vs gRPC vs GraphQL — pick based on client needs
  • Sync response vs async job + polling/webhook?
  • Auth model: token, session, API key?
  • Rate limiting strategy at API layer?
💡
APIs are a forcing function. Designing the interface first reveals hidden complexity — you'll discover fan-out, aggregation, or async needs before you're deep in architecture. Keep it lean: 3–5 endpoints is usually enough to demonstrate understanding.
D
Data Model & Storage
Pick the right stores, then sketch the schema
~12 min
Choose your stores
  • Relational (Postgres): complex joins, ACID transactions
  • Wide-column (Cassandra/DynamoDB): massive write throughput, time-series
  • Document (MongoDB): flexible schema, hierarchical data
  • Cache (Redis): hot reads, sessions, counters, pub-sub
  • Blob (S3): media, large files, backups
  • Search (Elasticsearch): full-text, faceted queries
Schema & access patterns
  • Define entities and their key fields
  • Identify primary access pattern → drives partitioning/index strategy
  • Denormalize if read-heavy; normalize if write-heavy
  • Pick partition key to avoid hot spots
  • Plan indexes for your top 2–3 query patterns
⚡
Start with the access pattern, not the entity. "What query is being served most?" determines schema far more than "what data exists." For NoSQL, your partition key is your most important design decision — think about distribution and hot partitions upfront.
I
Infrastructure & High-Level Design
Draw the system, then scale it
~18 min
Draw the happy path first
  • Client → LB → Service(s) → DB — get this on the board fast
  • Show data flow for the 1–2 most important features
  • Add a CDN/cache layer if read-heavy
  • Add a message queue if write-heavy or async processing needed
  • Show where async workers / consumers live
Scale each layer
  • Compute: horizontal scaling, stateless services
  • Data: replication (read replicas), sharding/partitioning
  • Cache: CDN for static, Redis for hot data, cache-aside vs write-through
  • Queue: Kafka (high throughput) vs SQS (managed, simpler)
  • LB: L4 vs L7, sticky sessions if stateful
Key architectural patterns
  • Fan-out on write vs fan-out on read (news feed, notifications)
  • CQRS — separate read/write models for high-scale reads
  • Event sourcing — when full audit trail matters
  • Saga pattern — distributed transactions across services
Failure & consistency
  • What happens when the DB is down? Circuit breakers?
  • Idempotency keys for retried writes
  • Eventual consistency — what's the lag tolerance?
  • Leader election for stateful components
🗺️
Diagram first, details second. Interviewers want to see your mental model of the whole system before you dive into any component. Draw boxes and arrows, narrate as you go, then drill into complexity. Use a queue anytime you see an impedance mismatch between producers and consumers.
O
Operability & Deep Dives
Show the system is production-ready and go deep
~14 min
Reliability & Resilience
  • Replication factor and failover strategy per store
  • Retry logic with exponential backoff + jitter
  • Bulkhead / circuit breaker between services
  • Graceful degradation — what works in a partial outage?
  • Multi-region: active-active vs active-passive?
Observability
  • Metrics: latency (p50/p99), error rate, throughput
  • Distributed tracing (correlation IDs across services)
  • Structured logging with sampling strategy
  • Alerts on SLO breach, not just raw errors
Security & Compliance
  • Authz: who can access what? (RBAC/ABAC)
  • Encryption at rest and in transit
  • Rate limiting & abuse prevention
  • PII handling — where is user data stored?
Interviewer-directed deep dives
  • Pick your 1–2 hardest unsolved problems and propose solutions
  • Respond to "what if traffic 10×s overnight?"
  • Explain trade-offs you made and what you'd do differently at larger scale
  • Know your bottlenecks and how to measure them
🎯
This is where senior engineers shine. Raise the hard problems yourself — don't wait for the interviewer. "The tricky part here is X, and here's how I'd handle it" is far stronger than being led there. Have 2–3 trade-off narratives ready.
